Internal Reference Document

Container Packing Optimizer
Rules & Stacking Logic

A complete reference for how pallets and boxes are grouped, stacked, ordered, and validated — including all constraints, edge cases, and what the system cannot handle.

Edelman — Logistics Optimization May 2026

1. Container Dimensions & Physical Limits

All calculations are based on a standard 40-foot high-cube shipping container with the following internal dimensions. Every constraint in the system is derived from these numbers.

Parameter Default Value What It Controls
Internal length1 203 cmHow many pallet rows fit end-to-end
Internal width235 cmHow many pallets fit side-by-side across
Internal height270 cmMaximum stack height inside the container
Door opening height250 cmMaximum height of the row nearest the door
Max payload weight18 000 kgCombined weight of all rows in one container
Gap between rows5 cmForklift clearance between consecutive pallet rows
Overridable via Environment Variables
All six values above are defaults only. Any of them can be changed by setting the corresponding environment variable on the server (CONTAINER_LENGTH_CM, CONTAINER_WIDTH_CM, etc.) without touching any code.

Each row spans the full 235 cm width. The last row (nearest the door) must be short enough to pass through the 250 cm opening.

2. Pallet Footprints — What Is Supported

The system recognises four pallet footprints. Any pallet whose dimensions do not match one of these (within a ±2 cm tolerance) is silently skipped and excluded from the optimisation. A warning is printed but the run continues.

Footprint (L × W) Typical label Pallets across 235 cm width Can rotate?
115 × 115 cm A2 2 pallets (2 × 115 = 230 cm ≤ 235 cm) No (square — rotation has no effect)
115 × 108 cm A1 2 pallets across (either orientation) Yes — both orientations tried
115 × 77 cm C2 / C3 2 pallets across (115 cm faces the width axis) Yes — both orientations tried
77 × 77 cm D2 3 pallets (3 × 77 = 231 cm ≤ 235 cm) No (square)
Not Supported
Any pallet footprint outside these four types — for example 100 × 100 cm or 120 × 80 cm — is dropped from the optimisation entirely. The run does not fail, but those pallets will not appear in any container. Always check the warnings in the output if your total pallet count seems lower than expected.
Known Gap — 115 × 115 cm pallets taller than 130 cm
The block-type table does not contain an entry for a 115 × 115 cm pallet with height > 130 cm. Such pallets are silently dropped. The equivalent tall entry exists only for 115 × 77 cm. This is a hardcoded limitation that would need a code change to fix.

3. Height Bands & Stacking Rules

Once the footprint is identified, each pallet is classified into a height band based on its individual pallet height. The band determines how many pallets can be stacked on top of each other, which in turn determines how many pallets form one row-block.

How Block Height Is Calculated
The actual block height is: stack count × tallest individual pallet height in that block. For example: 4 pallets of 60 cm each, stacked 2 high (89–130 band) → block height = 2 × 60 = 120 cm. The tallest pallet in the batch is used conservatively.

Why the 77 × 77 < 66 cm Band Has 9 Pallets
Small 77 × 77 pallets shorter than 66 cm would theoretically stack 4 high (as other types do), giving 12 pallets per block. However the block height would be 4 × 65 = 260 cm — very close to the container ceiling. The table conservatively caps it at 3 stacks (9 pallets) to match the safer 66–89 rule. This is hardcoded.

4. The Exact-Multiple Rule

This is the most common reason an order fails to run. Your order quantity for each pallet type must be an exact multiple of the pallets-per-block number for that type.

Rule
order quantity mod pallets_per_block = 0
If even one pallet type violates this, the entire run is rejected. No containers are packed. The error message tells you exactly how many pallets to add for each type.

Examples

Why This Rule Exists
A "row-block" is a physical unit: a set of pallets that form exactly one complete row across the container width. You cannot have half a row. If you ordered 7 pallets of a type that forms 8-pallet rows, there is no valid row to place — so the run refuses to continue rather than silently dropping a pallet.

5. Loading Order — The Door Height Rule

Rows inside a container must be arranged in non-increasing height order from the back of the container to the door. This is both a physical necessity (the door opening is 250 cm, smaller than the maximum container height of 270 cm) and a practical loading constraint.

Side view — mixed heights in one container: BACK DOOR (250 cm opening) ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌─────────┐ │ │ │ │ │ │ │ │ ▲ │ Row 1 │ │ Row 2 │ │ Row 3 │ │ Row 4 │ │ 250 cm │ 250 cm │ │ 240 cm │ │ 200 cm │ │ 180 cm │ ▼ │ │ │ │ │ │ │ │ └──────────┘ └──────────┘ └──────────┘ └─────────┘ 🚪 ◄──── loaded first, from the back ─────────────────────► ✅ Allowed: heights decrease (or stay equal) towards the door. ❌ Not allowed: a short row followed by a taller row closer to the door.

The Two-Category Split

Rows are internally classified as:

  • door_ok — block height ≤ 250 cm. Can be placed in any position, including the door position.
  • door_over — block height > 250 cm. Can only be placed at the rear, never at the door position.
What Happens When Both Types Are Present
The solver ensures that all door_over rows go at the rear and at least one door_ok row is placed at the door. If a future container will also need a door_ok row, the system reserves one — it will not use all available short rows in the current container if doing so would leave a future container without a valid door row.
Fatal Error: No Door-Compliant Row Available
If all remaining rows are taller than 250 cm and there are no shorter rows left to place at the door, the run fails with an error. This means it is physically impossible to close and seal the container. The fix is to either reduce pallet stack heights or split the order differently.

6. The Optimiser — What It Maximises

For each container pass, a mathematical optimiser (Google OR-Tools CP-SAT) chooses which row-blocks to load and in what order. It is not a simple greedy rule — it evaluates the problem as a whole within a time limit.

Objective (priority order)

  1. Maximise the number of pallets loaded — getting as much of your order into this container as possible takes absolute priority.
  2. Maximise the length used — as a tiebreaker, the solver prefers arrangements that leave less wasted space at the back.

Hard Constraints (cannot be violated)

ConstraintRule
LengthSum of all row lengths + (number of gaps × 5 cm) ≤ 1 203 cm
WeightSum of all row weights ≤ 18 000 kg
Height orderingEach row must be no taller than the row behind it (non-increasing, back → door)
Door heightThe last (door-side) row must have height ≤ 250 cm
No duplicatesEach row-block can appear in at most one container
ContiguityEmpty row slots must be at the end — you cannot have a gap in the middle of a container
Deterministic Results
The solver always produces the same result for the same input. It runs with a fixed random seed (42) and a single search thread. The time limit is 3 seconds per container — sufficient for typical orders. Very large or complex orders may benefit from a higher time limit (set via SOLVER_TIME_LIMIT_SEC).

Container Loop

The system repeatedly solves one container at a time until all row-blocks have been placed:

  1. Offer the remaining blocks to the solver.
  2. The solver picks the best subset that fits in one container.
  3. Those blocks are removed from the pool.
  4. Repeat until the pool is empty or 30 containers are reached.
30-Container Safety Limit
If the solver has not emptied the pool after 30 containers, the run is aborted. In practice this limit is never reached with real orders — hitting it usually indicates a problem with the input data.

7. NP (Loose) Box Rules

Rows in the Excel file marked with type code NP are treated as loose boxes — individual cartons rather than palletised loads. They are handled separately from pallet rows and have their own set of rules.

What IS Possible

  • ✓ Boxes are placed in the tail space — the unused length at the back of the container after all pallet rows.
  • ✓ Multiple box types share the same tail section (column-based packing — no wasted dedicated zones per type).
  • ✓ All six orientations of each box are tried to find the best fit.
  • ✓ Boxes are stacked in the full 235 cm width × up to 250 cm height of the tail zone.
  • ✓ Leftover boxes that don't fit in the first container carry over to subsequent containers.
  • ✓ Box weight contributes to the container weight budget.

What Is NOT Possible

  • ✗ Boxes cannot be placed on top of pallet rows — only pallets can sit on top of pallets.
  • ✗ Boxes cannot be interleaved between pallet rows — tail only.
  • ✗ If there is no tail space in any container, boxes cannot be placed at all (a warning is issued).
Container side view — NP boxes in the tail zone: BACK LAST ROW TAIL SPACE DOOR ┌────────┐ ┌────────┐ ┌────────┐ ┌──────────────┐ │ │ │ │ │ │ 5 cm │ [Box A][A] │ │ Row 1 │ │ Row 2 │ │ Row 3 │ gap │ [Box B][B] │ │ │ │ │ │ │ ←── │ [A][A][B][B] │ └────────┘ └────────┘ └────────┘ └──────────────┘ ◄────── pallet zone ──────────────────────►◄─── NP zone ──►
Column-Based Packing Algorithm
The tail is filled one column at a time along the length axis. Each column uses the full width and height of the tail opening (235 × 250 cm). Within a column, box types are stacked in horizontal strips from the floor up, tallest type first. The column depth equals the largest box depth among all types being placed, so every type fits at least one layer. The cursor then advances by that depth and a new column begins.

8. Fill Recommendations

After the main packing is complete, the system identifies free zones in each container and recommends what you could add to your order to fill them. Recommendations are advisory — they do not change the packing plan.

Two Free Zones per Container

ZoneWhere it isWhat can go thereCeiling height
Tail Unused length after the last pallet row Pallet blocks AND NP boxes 250 cm (door height)
Atop Headroom above each placed pallet row Pallet blocks only (no boxes on pallets) 250 cm minus row height

Proportional Allocation

Pallet type recommendations in the tail zone are distributed proportionally to your existing order quantities. If you ordered 3× as many A2 pallets as C2 pallets, the tail recommendation will suggest roughly 3× as many A2 as C2 additions. This avoids recommending a large quantity of a pallet type you barely use.

Recommendation Objectives
The default objective is min_leftover — maximise the space filled. Alternatives available via the RECOMMEND_OBJECTIVE environment variable: min_pallets (fill with as few extra pallets as possible), max_weight (fill heaviest first), max_value (fill highest-FOB-price items first — requires FOB prices in the Excel file).

9. Validation — Automatic Checks

Every result is checked against eight rules before any report is produced. Errors stop the run; warnings are shown in the UI but the report is still generated.

#CheckLevelDescription
1Block coverageERROR Every block that entered the solver must appear in exactly one container — no block dropped, none duplicated.
2Pallet countERROR Total pallets across all containers must equal total pallets from the input. Any discrepancy means pallets were lost or invented.
3Container over-lengthERROR The sum of row lengths plus gaps must not exceed 1 203 cm (1 cm rounding tolerance allowed).
4Door height violationERROR The last row in each container must have height ≤ 250 cm.
5Weight exceededERROR Loaded weight must not exceed 18 000 kg (0.5 kg rounding tolerance).
6Y-coordinate consistencyWARNING Each row's reported start position must equal the end of the previous row plus the 5 cm gap.
7NP box quantityWARNING More boxes placed than ordered → ERROR. Fewer boxes placed than ordered → WARNING ("unplaced boxes").
8Low fill on non-last containersWARNING A non-final container filled to less than 50% of its length is flagged as unusually low. This may indicate a data issue.
–Tall rows at rearWARNING Informational: reports which containers have rows taller than the door loaded from the back. These require rear-loading equipment.
–Zero weightWARNING If all containers report zero loaded weight, pallet weights are missing from the Excel — the weight constraint is effectively inactive.

Robustness Features

  • Blank header rows: the system scans the sheet to find the actual header row — files with 1–2 blank rows at the top are handled automatically.
  • Column name matching: prefix-based, case-insensitive, with whitespace normalisation. A column called "Ordered External Packaging Quantity" (double space) will still be found.
  • Decimal separator: both 1.15 (dot) and 1,15 (comma) are accepted in dimension strings.
  • Mixed sheets: pallet rows and NP box rows can coexist on the same sheet — they are separated by the type code.
